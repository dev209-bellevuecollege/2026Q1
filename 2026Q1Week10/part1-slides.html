Part 1: The Evolution of Programming

From Punch Cards to AI — How We Got Here

15 Minutes
Speaker Notes: Open with a question: "Who here has ever seen a punch card?" Show of hands. This section traces the arc of programming history to show that AI-assisted development is the natural next step in a pattern that's been repeating for 80 years. Each generation abstracts away the "how" and lets us focus more on the "what."

Speaker Notes: Walk through each era. The key insight: at every step, someone invented a tool that let the next generation of programmers think at a higher level. Assemblers translated mnemonics to machine code. Compilers translated C to assembly. Today, LLMs translate human language to code in any language. Ask: "What's the common thread?" Each tool handles the lower-level details so we can focus on the problem we're actually trying to solve.
gcc

The Compiler Analogy

AI as the Next-Generation Compiler

Traditional Compiler

You write C code. The compiler generates assembly and machine code.

// You write this:
int add(int a, int b) {
  return a + b;
}

// Compiler produces assembly:
// MOV eax, [esp+4]
// ADD eax, [esp+8]
// RET

AI "Compiler"

You describe what you want. The AI generates the implementation.

// You write this:
"Create a React component
that fetches user data from
/api/users and displays it
in a sortable table with
pagination."

// AI produces: UserTable.tsx
// 120 lines of working code

You trust the compiler to generate correct machine code. Can you trust AI the same way? Not yet. That's why review matters.

Speaker Notes: This analogy is powerful but has limits. Point out that compilers are deterministic — the same input always produces the same output. LLMs are stochastic — they can produce different output each time. Compilers have formal correctness proofs. LLMs have probabilities. This is why we still need to review AI output, and why understanding the fundamentals remains essential. You wouldn't trust a compiler that sometimes produces wrong code — and LLMs DO sometimes produce wrong code.
WHAT

Declarative Thinking

Specify What, Not How

You Already Think This Way

  • SQL: "Give me all users older than 25" — you don't specify HOW to scan rows
  • CSS Flexbox: "Make these items center-aligned in a row" — the browser handles the math
  • React: "Here's what the UI should look like given this state" — React figures out what changed
  • HTML: "This is a navigation list" — the browser decides how to render and make it accessible

AI Takes This Further

  • Spec: "Build a todo app with drag-and-drop reordering, localStorage persistence, and dark mode"
  • Result: Complete working application
  • Your role: Review, refine, direct — like an architect reviewing blueprints
  • Key shift: From writing every line to directing the solution

Declarative programming says "describe the result." AI-assisted development says "describe the problem."

Speaker Notes: Students already use declarative approaches every day! SQL doesn't require you to write a loop over database rows. CSS Flexbox doesn't require you to calculate pixel positions. React doesn't require you to manually diff the DOM. AI-assisted development is the next step in this same direction: you describe the problem or the desired outcome, and the system figures out the implementation. But just like SQL can produce slow queries and React can produce performance issues, AI can produce buggy or insecure code. The human still needs to understand what's happening underneath.
WHY

Closer to the Real Problem

From Mechanics to Meaning

The Real Question Was Always:

"What problem are we actually trying to solve for the user?"

Where Time Used to Go

  • 60% — Boilerplate & configuration
  • 20% — Debugging syntax & type errors
  • 10% — Searching Stack Overflow
  • 10% — Thinking about the actual problem

Where Time Can Go Now

  • 40% — Understanding the user's problem
  • 25% — Designing the right solution
  • 20% — Reviewing & refining AI output
  • 15% — Testing & validation

Technology exists to solve human problems. AI tools let us spend more time on the "why" and less on the "how."

Speaker Notes: This is the most important slide in Part 1. The promise of AI-assisted development isn't "write code faster." It's "spend more of your time on the things that actually matter." The mechanical parts of programming — setting up build systems, writing CRUD routes, configuring authentication — are increasingly automatable. What isn't automatable is understanding what the user actually needs, what problem is worth solving, and whether the solution actually works in the real world. That's where YOUR value lies.
