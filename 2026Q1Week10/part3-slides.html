CTX

What is "Context"?

The #1 Factor in AI Output Quality

Definition

Context is everything the AI can "see" when generating a response: your code, your prompt, project files, conversation history, and any rules or specifications you've provided.

Context Includes:

  • • Your current codebase files
  • • The conversation history so far
  • • Project configuration (package.json, etc.)
  • • Rules files (CLAUDE.md, .cursorrules)
  • • Error messages you share
  • • Screenshots and images

Context Window Limits

  • • Every AI has a maximum context window
  • • Ranges from 128K to 1M+ tokens
  • • More context = better results (usually)
  • • Too much context = noise and confusion
  • • Agentic tools manage this automatically
  • • Quality > Quantity

Better context = better output. The AI is only as good as what you give it to work with.

Speaker Notes: Think of context like a briefing for a new team member. If you say "fix the bug," they have no idea where to start. If you say "there's a null reference error in the UserProfile component on line 47 when the user has no avatar set; here's the error message and the relevant code," they can help immediately. The AI is the same way. Agentic tools like Claude Code automatically read relevant files from your codebase, which is why they produce better results than chat interfaces — they have more context. But even with agentic tools, you get better results when you provide clear, specific instructions.

Managing Context with Project Files

Tell the AI About Your Project Once

CLAUDE.md (Claude Code)

# Project: Student Portal
# Stack: React 19 + TypeScript
# Build: Vite

## Conventions
- Use functional components
- Use Tailwind CSS for styling
- Follow existing naming patterns
- Write tests for new features
- Use React Router for navigation

## Important
- API base URL: /api/v2
- Auth: JWT stored in httpOnly cookie
- Never use localStorage for tokens

.cursorrules (Cursor)

You are an expert in React
and TypeScript.

Key conventions:
- Prefer named exports
- Use CSS modules, not inline styles
- Components in PascalCase
- Hooks in camelCase with "use" prefix
- Always handle loading and error states
- Use Zod for runtime validation

Testing:
- Vitest + React Testing Library
- Test behavior, not implementation

Write it once, benefit every time. These files are persistent context — the AI reads them automatically on every interaction.

Speaker Notes: This is one of the most practical tips in the whole lecture. Every project should have one of these files. Claude Code reads CLAUDE.md automatically. Cursor reads .cursorrules. Codex reads AGENTS.md. The point is the same: you're telling the AI about your project's conventions, tech stack, and constraints ONCE, and it applies that knowledge to every interaction. Without these files, you'd have to repeat "use Tailwind, not CSS modules" or "we use React Router, not Next.js routing" every single time. Show the students: it takes 5 minutes to create, saves hours over the life of a project.
TOOL

The AI Toolbox

Slash Commands, Skills, MCP & Sub-Agents

Slash Commands

Quick actions you invoke by typing / in the AI tool.

  • /init — Generate a CLAUDE.md for the project
  • /clear — Reset conversation context
  • /compact — Summarize & compress context
  • /review — Review code changes

Think of these as keyboard shortcuts for common AI workflows.

Skills

Pre-built, specialized prompts that package expertise into a single command.

  • • Triggered via slash commands (e.g., /commit)
  • • Inject a detailed system prompt with best practices
  • • Ensure consistency — same quality every time
  • • Can be custom-built for your team's workflows

A skill = a reusable prompt template with tool access baked in.

MCP (Model Context Protocol)

An open standard that lets AI tools connect to external services: databases, APIs, GitHub, Jira, Figma, your own tools. Like USB for AI — plug in any data source and the AI can read from it.

Sub-Agents

The AI spawns child agents to handle parallel tasks. The main agent coordinates while sub-agents research, explore code, or execute independently — then report back. Divide and conquer.

Speaker Notes: These four primitives are the building blocks of modern AI development tools. Slash commands are the simplest — quick actions like /init to bootstrap a project or /clear to reset context when you're starting a new task. Skills are the next level: they're specialized prompts that encode expertise. When you type /commit, you're not just asking the AI to commit; you're invoking a skill that knows how to write good commit messages, check for sensitive files, and follow your project's conventions. MCP is the most powerful concept here: it's how AI tools connect to the outside world. Without MCP, the AI can only see your files. With MCP, it can query your database, check your Jira board, pull designs from Figma. It's an open protocol, meaning any tool can implement it. Sub-agents are about parallelism: instead of doing everything sequentially, the AI can spin up workers. "You go research the API docs, you go find similar patterns in our codebase, you go check the test coverage" — all at the same time.
SYS

How They Work Together

Context Flow & the Right Tool for the Job

CLAUDE.md + MCP Servers + Your Prompt + Codebase Files → Context Window

How Each Affects Context

  • Slash Cmds — Manage context (/clear resets, /compact compresses)
  • Skills — Inject context (add specialized prompts and instructions)
  • MCP — Expand context (connect to external data sources)
  • Sub-Agents — Protect context (offload work so the main window stays focused)

Why Context Management Matters

The context window is finite. Fill it with noise and quality drops. Sub-agents do heavy exploration in their own window and return only the answer, keeping your main context clean.

When to Use Each

  • Slash Commands — Quick actions. Starting a task, committing, reviewing, clearing stale context.
  • Skills — Repeatable workflows. Standardizing how your team writes commits, reviews PRs, or grades assignments.
  • MCP — When the AI needs data it can't get from your files: issue trackers, design tools, databases, documentation sites.
  • Sub-Agents — Complex tasks with independent parts. "Research this API, refactor that module, and write tests" — all in parallel.
Speaker Notes: Draw the analogy to a professional kitchen. Slash commands are like the quick-access tools on your belt — knife, tongs, thermometer. Skills are your recipe cards — proven procedures that produce consistent results. MCP is your supply chain — connections to the pantry, the fridge, the delivery service. Sub-agents are your sous chefs — you delegate prep work so you can focus on the main dish. The context window diagram at the top is key: everything flows into one shared space. If you fill it with irrelevant file contents from a broad search, the AI loses focus. That's why sub-agents are so valuable — they do the messy exploration in their OWN context and report back a clean summary. Practical tip: if the AI starts giving worse answers after a long conversation, use /compact or /clear to reset. Context pollution is real.

Without a Spec

"Build me a todo app"

Result: generic, missing features, wrong assumptions, needs 5 rounds of "no, I meant..."

With a Spec

"Build a todo app with: drag-and-drop reorder, due dates with overdue highlighting, localStorage persistence, dark mode toggle, keyboard shortcuts (Enter to add, Delete to remove), and a 'clear completed' button."

Result: exactly what you asked for, first try.

Speaker Notes: The spec-driven process is how professional software teams have always worked — AI tools just make it even more important. When you give vague instructions, you get vague results. When you give precise specs, you get precise implementations. Walk through the example: "Build me a todo app" could mean a hundred different things. The detailed spec leaves nothing to interpretation. This doesn't mean you need a 50-page document — even a bullet list of 5-10 requirements dramatically improves output quality. The AI can also help you WRITE the spec: "Help me think through what a todo app needs" is a great first prompt.
LANG

Writing Precise Specifications

Human Language as a Programming Language

The better you write, the better the AI codes. Precision in language = precision in output.

A Good Spec Includes:

  • What — The feature or behavior you want
  • Who — The user and their goal
  • Constraints — Tech stack, patterns, limits
  • Edge cases — What happens when things go wrong?
  • Examples — Sample inputs and expected outputs
  • Non-goals — What you explicitly don't want

Precision Techniques:

  • Be specific: "3 columns" not "a few columns"
  • Give examples: "Like GitHub's issue list"
  • State constraints: "No external libraries"
  • Define boundaries: "Max 100 items in the list"
  • Specify behavior: "On error, show a toast notification that auto-dismisses after 3 seconds"
  • Use non-goals: "Don't add authentication"
Speaker Notes: This is a skill that transcends AI tools. Writing precise specifications is the same skill as writing good user stories, bug reports, or technical documents. It's a communication skill. Emphasize that "non-goals" are incredibly valuable — they prevent the AI from over-engineering. If you say "build a search feature" without non-goals, the AI might add full-text search with fuzzy matching and autocomplete when all you needed was a simple filter. Also: examples are powerful. Saying "the table should look like GitHub's issues list" gives the AI a visual reference it can reason about.
STOP

Common Anti-Patterns

What NOT to Do

Don't

  • Blindly accept output — Always read what was generated. AI makes subtle mistakes (wrong API endpoints, missing error handling, security flaws).
  • Use AI for what you don't understand — If you can't verify the output, you can't catch the bugs. Learn the fundamentals first.
  • Skip testing — AI-generated code needs the same testing as human-written code. Maybe more.
  • Ignore the conversation — If the AI asks clarifying questions, answer them. It's trying to give you better context.

Do

  • Start with a plan — Use "plan mode" or ask the AI to outline its approach before coding.
  • Iterate in small steps — Build feature by feature, not everything at once. Verify each step.
  • Provide feedback — "That's close, but change X" is more effective than starting over.
  • Use version control — Commit working states so you can roll back if the AI breaks something.

Remember from Week 1: "If you can't explain what the AI wrote, you haven't finished the task."

Speaker Notes: Call back to Week 1's "audit rule." These anti-patterns are extremely common, especially among developers new to AI tools. The most dangerous one is "use AI for what you don't understand." If someone asks an AI to implement authentication and doesn't understand JWT tokens or session management, they won't catch a critical security flaw. That's how vulnerabilities get shipped. The "iterate in small steps" principle is crucial: instead of "build me an entire e-commerce site," start with "create the product listing page" and verify it works before moving on.

The AI Output Review Checklist

Before You Accept, Verify

Correctness

  • ☐ Does it do what I asked?
  • ☐ Does it handle edge cases?
  • ☐ Does it match our existing patterns?
  • ☐ Did it introduce any regressions?

Quality

  • ☐ Is the code readable?
  • ☐ Is it over-engineered?
  • ☐ Are there unnecessary dependencies?
  • ☐ Would I approve this in a code review?

Security (Week 9 Review!)

  • ☐ Any XSS vulnerabilities?
  • ☐ Is user input sanitized?
  • ☐ Are secrets kept server-side?
  • ☐ Auth tokens stored securely?

Understanding

  • ☐ Can I explain every line?
  • ☐ Do I understand WHY this approach?
  • ☐ Could I modify it if requirements change?
  • ☐ Have I learned something new?
Speaker Notes: This checklist ties together everything from the course. Security from Week 9. Code quality from throughout the course. Understanding from the audit rule in Week 1. The "understanding" section is the most important for students: if the AI teaches you something new, great! But if you accept code you don't understand, you're accumulating "understanding debt" that will bite you later. This checklist should become second nature. Print it out, stick it next to your monitor.
