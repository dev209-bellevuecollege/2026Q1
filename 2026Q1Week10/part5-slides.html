Demo 1: Claude Code

Building a Feature from a Spec

The Demo Scenario

We'll use Claude Code to build a weather dashboard widget — a React component that fetches weather data and displays it with a clean UI. We'll start with a spec and watch the AI build it step by step.

What to Watch For:

1

How Claude Code reads the codebase first to understand the project structure

2

How it asks clarifying questions or proposes a plan before coding

3

How it creates and edits multiple files (component, styles, tests)

4

How it runs commands to verify the code works (build, test, lint)

The Spec We'll Use:

# Weather Widget Spec

Create a WeatherWidget React component:

Requirements:
- Fetch weather from a public API
- Display temperature, conditions, icon
- Show a 3-day forecast
- Responsive: card layout on mobile,
  inline on desktop
- Loading skeleton while fetching
- Error state with retry button

Constraints:
- Use existing project conventions
- No new dependencies
- Accessible (ARIA labels)
- Dark mode compatible
Speaker Notes: Before starting the demo, walk through the spec on the right side. Point out how it follows the principles from Part 3: clear requirements, specific constraints, explicit about what we want and don't want ("no new dependencies"). Then open the terminal, navigate to a React project, and run Claude Code. Type or paste the spec. Narrate as Claude Code works: "Watch — it's reading the project structure first... now it's looking at our existing components to match the style... now it's creating the component file..." If it asks a question, answer it to show the interactive nature. Key teaching moment: point out how the CLAUDE.md file (if present) automatically provides project context.

Claude Code: Key Moments to Note

What Makes This Different from Chat

Agentic Behavior

  • • Reads files autonomously — it decides what to look at based on your request
  • • Creates a plan — outlines what it'll do before doing it
  • • Runs tests — verifies its own output
  • • Self-corrects — if tests fail, it reads the error and fixes the code
  • • Commits changes — creates a git commit with a descriptive message

Your Role During the Demo

  • • Approve file edits — you control what gets written
  • • Answer questions — the AI may ask for clarification
  • • Course correct — "That's not what I meant, try X instead"
  • • Review the output — check for issues before accepting
  • • Iterate — "Now add a settings gear icon"

Notice: you never opened a file or typed any code. But you directed every decision. That's the new workflow.

Speaker Notes: This slide is meant to be shown AFTER the demo, not before. Use it to debrief. Walk through what students just saw: "Did you notice how it read our existing components first? It matched our naming conventions automatically because of the CLAUDE.md file." Highlight the human-in-the-loop moments: "I approved each file edit. I could have rejected any of them. When it used the wrong API endpoint, I corrected it and it adapted." The key insight: "I didn't write code, but I made all the important decisions. That's the 'director' role we talked about in Part 4."

Demo 2: Google Antigravity

The Manager View & Multi-Agent Workflow

The Demo Scenario

We'll explore Antigravity's two modes: the Editor View for traditional AI-assisted coding, and the Manager View where we'll assign tasks to multiple agents working in parallel.

Editor View (Synchronous)

  • • Familiar IDE experience
  • • AI tab completions
  • • Inline chat commands
  • • Similar to Cursor or Copilot

Artifacts

Agents produce verifiable outputs: task lists, implementation plans, screenshots, and browser recordings you can review.

Manager View (Asynchronous)

  • • Control center for multiple agents
  • • Assign different tasks to each agent
  • • Agents work in parallel workspaces
  • • Review results when they finish
  • • Like being the PM of an AI team

Demo Tasks

Agent 1: "Build the login page"
Agent 2: "Create the API routes"
Agent 3: "Write the test suite"

Speaker Notes: This demo shows a fundamentally different paradigm: you're not pair-programming with one AI — you're managing a TEAM of AIs. Open Antigravity, show both views. In the Editor View, do a quick inline task to show students the Cursor-like experience. Then switch to Manager View. Create 3 agents and assign each a different task. While they work, explain: "Right now, three agents are working simultaneously on three different parts of the project. I'm going to review their work when they finish." Show the artifacts: "This agent produced a task list showing its plan. This one took a screenshot of the UI it built." Key moment: review the code from each agent and show how you'd merge their work. Point out the trust-but-verify workflow.

Your First Exercise

Pick any tool above and try this prompt:

"Create a React component called CountdownTimer that:
- Accepts a targetDate prop (ISO string)
- Shows days, hours, minutes, seconds remaining
- Updates every second
- Shows 'Time's up!' with confetti animation when done
- Is responsive and accessible
- Uses no external dependencies"
Speaker Notes: Give students concrete next steps. Walk through the installation commands for at least one tool. The exercise prompt is designed to be achievable but non-trivial — it involves state, effects, accessibility, and animation. It's a great test of how well the AI handles a real-world component spec. Encourage students to try the SAME prompt on two different tools and compare the output. That's the best way to understand tool differences. If time permits, have students try it in class right now. Also mention: these tools are free or have free tiers. There's no excuse not to try them.
