fn

Function Components

The Modern Way to Build React

A Component is a Function

// A simple component
function Greeting() {
  return (
    <div>
      <h1>Hello, World!</h1>
      <p>Welcome to React</p>
    </div>
  );
}

// Using the component
<Greeting />

Component Rules

✓ Must Follow

  • • Name starts with Capital letter
  • • Returns JSX (or null)
  • • Returns single parent element

✗ Common Mistakes

  • • greeting() - lowercase name
  • • Greeting() - calling as function
  • • Forgetting to return JSX

Components are instantiated with JSX syntax: <Greeting /> NOT Greeting()

logic

Components with Logic

JavaScript Before the Return

Adding Logic

function UserCard() {
  // Variables and logic go HERE
  const name = "Alice";
  const age = 25;
  const isAdult = age >= 18;

  const greeting = isAdult
    ? "Welcome!"
    : "Hi there!";

  // JSX uses those variables
  return (
    <div className="card">
      <h2>{name}</h2>
      <p>Age: {age}</p>
      <p>{greeting}</p>
    </div>
  );
}

The Pattern

1. Define variables & logic
↓
2. Calculate derived values
↓
3. Return JSX using those values

Tip: Keep the return statement clean! Move complex logic before it.

🛠

Component Composition

Components Inside Components

Building a Page

function Header() {
  return <header>My Site</header>;
}

function Footer() {
  return <footer>© 2026</footer>;
}

function App() {
  return (
    <div>
      <Header />
      <main>
        <h1>Welcome</h1>
        <p>Page content...</p>
      </main>
      <Footer />
    </div>
  );
}

Component Tree

App

├ Header

├ main

└ Footer

Benefits

  • • Reuse Header/Footer anywhere
  • • Easy to understand structure
  • • Change one place, updates everywhere
<>

Fragments

Grouping Without Extra DOM Elements

The Problem

// This adds an extra <div> to the DOM
function UserInfo() {
  return (
    <div>  // Extra wrapper!
      <h2>Alice</h2>
      <p>Developer</p>
    </div>
  );
}

Extra divs can break CSS layouts (flexbox, grid) and add unnecessary DOM nodes.

The Solution: Fragments

// Using Fragment syntax
function UserInfo() {
  return (
    <>  // No extra DOM element!
      <h2>Alice</h2>
      <p>Developer</p>
    </>
  );
}

// Or use React.Fragment
<React.Fragment>
  ...
</React.Fragment>

<>...</> is shorthand for <React.Fragment> - groups children without adding DOM elements

📁

File Organization

One Component Per File

Project Structure

src/ |-- main.jsx # Entry point |-- App.jsx # Root component |-- index.css # Global styles +-- components/ # Your components |-- Header.jsx |-- Header.css # Component styles |-- Footer.jsx |-- Card.jsx +-- Button.jsx

Export & Import

// Header.jsx
function Header() {
  return <header>...</header>;
}

export default Header;


// App.jsx
import Header from './components/Header';

function App() {
  return (
    <div>
      <Header />
      ...
    </div>
  );
}

Convention: Component names = PascalCase, file names = match component name

old

Class Components (Legacy)

What You Might See in Older Code

Old Way: Class Components

import React from 'react';

class Greeting extends React.Component {
  render() {
    return (
      <h1>Hello World</h1>
    );
  }
}

Note: You'll see this in older tutorials, Stack Overflow, and legacy codebases.

Modern Way: Function Components

function Greeting() {
  return (
    <h1>Hello World</h1>
  );
}

Why Function Components?

  • • Simpler syntax
  • • Hooks (state management)
  • • Better performance
  • • React team recommendation

Rule: Always use function components for new code!
