Why HTTPS Matters

HTTP is Plaintext — Anyone Can Read It

HTTP: The Problem

With plain HTTP, all data travels as readable text across the network.

  • • Passwords sent in the clear
  • • Credit card numbers exposed
  • • API tokens visible to attackers
  • • Anyone on the same Wi-Fi can intercept

Think about coffee shop Wi-Fi — every request and response is readable by anyone on that network.

HTTPS: The Solution

HTTPS = HTTP + TLS encryption

  • ✓ All data is encrypted in transit
  • ✓ Server identity is verified
  • ✓ Data integrity is guaranteed
  • ✓ Required for modern web features

Browser Warning

Chrome and Firefox now flag HTTP sites as "Not Secure" in the address bar.

How TLS Works (Simplified)

The Handshake That Secures Every Connection

Two Types of Encryption

Asymmetric (Public/Private Key)

Two keys: public key encrypts, private key decrypts. Used during the handshake to exchange secrets safely.

Symmetric (Shared Secret)

One shared key for both encryption and decryption. Much faster. Used for all data after the handshake.

The TLS Handshake

1. Browser: "Hello, I want a secure connection"
↓
2. Server sends its TLS certificate + public key
↓
3. Browser verifies cert with Certificate Authority
↓
4. Both sides agree on a shared encryption key
↓
5. All data encrypted with shared key
CA

Certificates & Chain of Trust

How Browsers Know Who to Trust

What is a TLS Certificate?

A digital document that contains:

  • • The domain name it's issued for
  • • The server's public key
  • • Who issued it (Certificate Authority)
  • • Expiration date
  • • Digital signature proving authenticity

Try It Now

Click the lock icon in your browser's address bar to inspect any site's certificate.

The Chain of Trust

Root CA (pre-installed in your OS/browser)
↓ signs
Intermediate CA
↓ signs
Your Site's Certificate

Your browser comes with ~150 trusted Root CAs pre-installed. If the certificate chains back to one of these, the browser shows the lock icon.
