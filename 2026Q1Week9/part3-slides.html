CSP

What is CSP?

An HTTP Header That Controls What Browsers Allow

The Concept

CSP is an HTTP response header that tells the browser: "Only load resources from these approved sources."

If an XSS attack injects a script, the browser blocks it because it's not on the approved list.

Without CSP

Browser loads and runs anything on the page — including injected scripts.

How It Works

Server sends CSP header with response
↓
Browser reads the policy rules
↓
Approved resources load normally
↓
Unapproved resources are BLOCKED

Blocked resources appear as errors in the DevTools console — you'll see exactly what was stopped.

Speaker Notes: Analogy: "Imagine a bouncer at a club with a guest list. CSP is that guest list. The server says 'only let in scripts from my domain and Google Analytics.' Everything else gets turned away at the door." Key point: CSP doesn't prevent the XSS injection from happening — it prevents the injected code from EXECUTING. It's a safety net. Show in DevTools: if you visit a site with CSP and open the console, you'll see blocked resource warnings.
DIR

Key CSP Directives

Controlling Each Type of Resource

Directive Controls Example
default-src Fallback for all resource types default-src 'self'
script-src JavaScript files & inline scripts script-src 'self' cdn.example.com
style-src CSS files & inline styles style-src 'self' 'unsafe-inline'
img-src Images img-src 'self' images.unsplash.com
connect-src fetch(), XHR, WebSocket connections connect-src 'self' api.example.com
font-src Web fonts font-src 'self' fonts.gstatic.com
frame-src iframes frame-src youtube.com

Pro tip: Start with default-src 'self' (block everything external), then add exceptions as needed.

Speaker Notes: Walk through the table. "default-src is your baseline — set it to 'self' meaning only your own domain. Then override specific types as needed." For script-src: "This is the most important one for XSS prevention. If you set script-src 'self', no inline scripts or external scripts can run unless they're from your domain." For connect-src: "This controls where your fetch() calls can go — it's related to CORS which we'll cover next." Ask students: "What would happen if you set img-src 'self' but your app loads images from Unsplash?" (They'd be blocked!)
VAL

CSP Values & Keywords

The Building Blocks of Your Policy

'self'

Only allow resources from the same origin (protocol + domain + port).

'unsafe-inline'

Allow inline <script> and <style> tags. Defeats most XSS protection! Avoid if possible.

'unsafe-eval'

Allow eval(), new Function(), etc. Very dangerous — rarely needed.

'none'

Block this resource type entirely. E.g., frame-src 'none' blocks all iframes.

Domain Allowlisting

Specify exact domains to allow:

script-src 'self'
  cdn.jsdelivr.net
  *.google-analytics.com

'nonce-{random}' (Best Practice)

Allow specific inline scripts with a one-time token:

// Header
script-src 'nonce-abc123'

// HTML - only THIS script runs
<script nonce="abc123">
  // This is allowed
</script>

// Injected scripts have no nonce
<script>evil()</script>
// BLOCKED!
Speaker Notes: Key insight: 'unsafe-inline' has "unsafe" in the name for a reason! If you allow inline scripts, you've basically turned off XSS protection for scripts. "But wait — don't many apps use inline scripts?" Yes, and that's what nonces are for. A nonce is a random value generated on each page load. The server puts the nonce in the CSP header AND on the script tag. Only scripts with the matching nonce run. Injected scripts won't have the nonce, so they're blocked. For students: "In practice, React apps serve from a bundled JS file, not inline scripts, so 'self' is usually enough."
USE

CSP in Practice

Adding CSP to Your Projects

Method 1: HTTP Header (Best)

// Server sets this header:
Content-Security-Policy:
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' api.example.com;
  font-src 'self' fonts.gstatic.com

Method 2: Meta Tag

<!-- In your HTML <head> -->
<meta
  http-equiv="Content-Security-Policy"
  content="default-src 'self';
    script-src 'self';
    style-src 'self' 'unsafe-inline'"
>

A Starter Policy for React Apps

// Good starting point:
Content-Security-Policy:
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: blob:;
  connect-src 'self'
    https://your-api.com;
  font-src 'self';

Report-Only Mode (Testing)

Use Content-Security-Policy-Report-Only to log violations without blocking. Great for testing before enforcing.

Demo

Open DevTools → Console → Look for CSP violation warnings on any major website (GitHub, Twitter).
