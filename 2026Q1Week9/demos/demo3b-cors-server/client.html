CORS Client + Server Demo

Two servers, two origins — see CORS blocking and two ways to fix it

Client Origin: http://localhost:3000
API Origin: http://localhost:3001
Different ports = different origins. Requests from :3000 to :3001 are cross-origin.

Test the Three Approaches

1. No CORS Headers

http://localhost:3001/api/data
The API returns JSON but doesn't set any CORS headers. The browser will block our code from reading the response.

2. With CORS Header

http://localhost:3001/api/data-cors
Same API, same data — but this endpoint adds the Access-Control-Allow-Origin header. One line fixes it.

3. Same-Origin Proxy

/api/proxy
Our client server proxies the request. The browser sees a same-origin request, so no CORS involved at all.

Server Code — What Changed?

Endpoint 1: No CORS headers (blocked)
app.get('/api/data', (req, res) => { res.json({ message: 'Hello from the API server!', timestamp: new Date().toISOString() }); });
Endpoint 2: With CORS header (allowed) — one line added
app.get('/api/data-cors', (req, res) => { res.set('Access-Control-Allow-Origin', 'http://localhost:3000'); res.json({ message: 'Hello from the API server!', timestamp: new Date().toISOString() }); });
Endpoint 3: Proxy on the client server (same origin)
app.get('/api/proxy', async (req, res) => { // Server-to-server fetch — no browser, no CORS const response = await fetch('http://localhost:3001/api/data'); const data = await response.json(); res.json(data); });

Proxy Approach: Pros & Cons

Pros

  • Works with ANY API, even ones you don't control
  • Hides API keys from the browser
  • No CORS headers needed on the API
  • Can transform or cache responses

Cons

  • Adds latency (extra network hop)
  • Your server handles all the traffic
  • More infrastructure to maintain
  • If the CORS header fix is available, it's simpler
