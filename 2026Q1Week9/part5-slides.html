Your Frontend Code is Public

Everything Ships to the Browser

The Reality

Your JavaScript, HTML, and CSS are downloaded to every user's browser. Anyone can:

  • • View Source (right-click any page)
  • • Read all JS in DevTools Sources tab
  • • Search for strings like "API_KEY"
  • • Inspect network requests for tokens

Minification is NOT Security

Minified code is harder to read but trivial to reverse. Build tools like Source Map Explorer can reconstruct your original code.

Things You Should NEVER Put in Frontend Code

// NEVER do any of these!

// API secret keys
const STRIPE_SECRET = 'sk_live_...';

// Database connection strings
const DB_URL = 'postgres://user:pass@...';

// Private API tokens
const ADMIN_TOKEN = 'eyJhbG...';

// Third-party secret keys
const AWS_SECRET = 'wJalrXU...';

OK in Frontend

Public keys (Stripe publishable key, Google Maps API key with domain restrictions) are fine — they're designed to be public.

Speaker Notes: DEMO: Open any popular website. Open DevTools → Sources tab. Show students how you can see all the JavaScript files. Search for "api" or "key" in the source files. "Everything here was sent to your browser. If a secret is in this code, it's not secret anymore." Students often think minification = security. Show them: "This is minified code. But look — I can still search for strings. And with a source map, I can see the original code." The distinction between public and secret keys is important. Stripe's publishable key (pk_) is designed to be in frontend code. The secret key (sk_) must NEVER be.

.env Files in React Are NOT Secret

A Common and Dangerous Misconception

The Misconception

"I put my API key in .env so it's safe."

# .env file
VITE_API_KEY=super_secret_key_123
VITE_API_URL=https://api.example.com

What Actually Happens

The build tool copies the value directly into your JavaScript bundle:

// Your code:
fetch(import.meta.env.VITE_API_KEY)

// After build, your bundle contains:
fetch("super_secret_key_123")

// The value is literally in the file!

The Right Way: Backend Proxy

Keep secrets on your server. Frontend calls YOUR server, which adds the secret key:

React: fetch('/api/weather?city=NYC')
↓
Your Server: adds SECRET_KEY to request
↓
Server: calls weather API with key
↓
Returns data to React (no key exposed)

What .env IS Good For

Keeping values out of git (.gitignore) and varying between environments (dev vs production URLs). Not for actual secrets in frontend code.

Speaker Notes: This is one of THE most common mistakes students make. "Raise your hand if you've put an API key in a .env file in a React project." (Many will.) "That .env file keeps the key out of your git repository — great! But at BUILD TIME, Vite (or Create React App) literally copies the string into your JavaScript bundle. Anyone can see it." DEMO: Build a Vite/React project that uses an env variable. Then open the built JS file in dist/ and search for the env value — it's right there in plain text. Show the proxy pattern: "Your frontend calls YOUR backend. Your backend has the real secret and calls the third-party API."

npm Supply Chain Attacks

When Your Dependencies Turn Against You

The Problem

A typical React project has hundreds of npm dependencies. Each one is code written by strangers that runs with full access to your project.

Attack Type: Typosquatting

Attacker publishes packages with names similar to popular ones:

  • lodash → 1odash
  • express → expresss
  • react-router → react-rounter

Attack Type: Account Takeover

Attacker gains control of a legitimate package maintainer's account and pushes malicious code.

Case Study: event-stream (2018)

A popular npm package (2M weekly downloads) was handed to a new maintainer who added code that stole Bitcoin wallet keys.

  • • Affected thousands of projects
  • • Malicious code was hidden in a dependency
  • • Went undetected for 2 months

How to Protect Yourself

  • ✓ Run npm audit regularly
  • ✓ Check package download counts & age
  • ✓ Use package-lock.json (locks versions)
  • ✓ Read the package's GitHub repo before installing
  • ✓ Fewer dependencies = smaller attack surface
Speaker Notes: DEMO: Open a React project. Run ls node_modules | wc -l to show how many packages are installed (often 800+). "Every one of these ran code on your machine when you did npm install. Some run postinstall scripts automatically." Then run npm audit to show any known vulnerabilities. Tell the event-stream story in detail: "A popular package maintainer got tired of maintaining it. A new person offered to take over. They seemed legit. A few months later, they added a dependency that specifically targeted a Bitcoin wallet app called Copay. It stole wallet keys and sent them to the attacker." Key lesson: "Be skeptical of packages. Do you really need that left-pad library, or can you write the function yourself?"
