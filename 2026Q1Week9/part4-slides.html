SOP

The Same-Origin Policy

The Foundation of Web Security

What is an "Origin"?

An origin is the combination of:

Protocol + Domain + Port 
https://example.com:443/page

Same or Different Origin?

https://a.com vs https://a.com/page✓ Same
http://a.com vs https://a.com✗ Different
a.com vs api.a.com✗ Different
a.com:3000 vs a.com:5000✗ Different

The Same-Origin Policy Says:

"JavaScript on one origin cannot read data from a different origin."

This prevents evil.com from reading your bank data at bank.com using fetch().

Why Does This Exist?

Without it, any website could:

  • ✗ Read your Gmail inbox
  • ✗ Check your bank balance
  • ✗ Access your social media DMs
  • ✗ Fetch data from internal company APIs
Speaker Notes: This is foundational. Spend a moment on the origin definition. "Protocol + domain + port. ALL THREE must match." Walk through the examples table — subdomains are different origins, different ports are different origins. "Why does this matter? Imagine you're logged into your bank. You visit evil.com. Without the Same-Origin Policy, evil.com's JavaScript could do fetch('https://mybank.com/api/balance') and read the response — because your browser would send your bank cookies with the request." The SOP prevents evil.com from READING the response. The request still goes through (important for CORS understanding).
CORS

What CORS Actually Does

Controlled Exceptions to the Same-Origin Policy

CORS = Cross-Origin Resource Sharing

A system that lets servers explicitly allow certain cross-origin requests.

The server decides who's allowed in, not the browser.

The Key Idea

When your frontend (localhost:3000) fetches from your API (localhost:5000), that's cross-origin. The API must explicitly allow it.

The CORS Flow

Browser: fetch('https://api.example.com/data')
↓
Browser adds Origin: https://myapp.com header
↓
Server responds with Access-Control-Allow-Origin
↓
Browser checks: does the header match my origin?
↓
Yes → JS gets the data
No → CORS Error!

Key insight: CORS errors happen in the browser, not the server. The request actually reaches the server — the browser just refuses to show you the response.

Speaker Notes: This is the #1 misconception. "The server didn't block your request! The request went through, the server processed it, and sent a response. But the BROWSER looked at the response headers and said 'nope, the server didn't say you're allowed to read this.'" Draw the parallel to CSP: "CSP is the server telling the browser what to LOAD. CORS is the server telling the browser what other origins can READ its responses." Key example for students: "Remember when you ran React on localhost:3000 and your API on localhost:5000? That's cross-origin! Different ports = different origins."
PRE

Simple vs. Preflight Requests

Why Some Requests Need Permission First

Simple Request

Goes directly — no preflight needed.

  • ✓ Method: GET, HEAD, or POST
  • ✓ Only simple headers (Accept, Content-Type*)
  • ✓ Content-Type: text/plain, multipart/form-data, or application/x-www-form-urlencoded
// This is a "simple" request
fetch('https://api.example.com/data')

// So is this
fetch('https://api.example.com/data', {
  method: 'POST',
  headers: {
    'Content-Type': 'text/plain'
  }
})

Preflight Request (OPTIONS)

Browser sends an OPTIONS request first to ask permission.

Triggered when you use:

  • ✗ Methods: PUT, DELETE, PATCH
  • ✗ Custom headers (Authorization, X-*)
  • ✗ Content-Type: application/json
// This triggers a PREFLIGHT
fetch('https://api.example.com/data', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer token'
  }
})

Surprise: application/json triggers a preflight! This catches almost every modern API call.

Speaker Notes: "Here's why this matters: almost every API call you make with React sends Content-Type: application/json and an Authorization header. Both of those trigger a preflight." Show in DevTools Network tab: when making a cross-origin fetch with JSON, you'll see TWO requests — first an OPTIONS request, then the actual request. "The OPTIONS request is the browser asking 'hey server, is this origin allowed to send this kind of request?' If the server says yes, the real request goes through." Common gotcha: "If your server doesn't handle OPTIONS requests, every API call with JSON will fail with a CORS error."
HDR

CORS Headers Explained

What the Server Sends Back

Response Header Purpose Example
Access-Control-Allow-Origin Which origins can read responses https://myapp.com
Access-Control-Allow-Methods Which HTTP methods are allowed GET, POST, PUT, DELETE
Access-Control-Allow-Headers Which request headers are allowed Content-Type, Authorization
Access-Control-Allow-Credentials Allow cookies/auth headers true
Access-Control-Max-Age Cache preflight result (seconds) 86400

Example Server Response Headers

Access-Control-Allow-Origin: https://myapp.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 86400
Speaker Notes: Walk through each header. Allow-Origin is the most important — it must match the requesting origin exactly (or be *). Allow-Methods lists what HTTP methods are allowed. Allow-Headers lists what custom headers the client can send. Allow-Credentials is needed if you're sending cookies — but if this is true, Allow-Origin CANNOT be * (we'll cover this next). Max-Age caches the preflight response so the browser doesn't send OPTIONS every single time. "86400 = 24 hours. This is a nice performance optimization."
ERR

The CORS Error You've Seen

And How to Actually Fix It

Access to fetch at 'https://api.example.com/data' from origin
'http://localhost:3000' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the
requested resource.

Fix 1: Configure Your Server

// Express.js with cors package
const cors = require('cors');
app.use(cors({
  origin: 'http://localhost:3000',
  credentials: true
}));

Fix 2: Dev Proxy (React/Vite)

// vite.config.js
server: {
  proxy: {
    '/api': 'http://localhost:5000'
  }
}

Wrong Fix: Allow All Origins

// DON'T do this in production!
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
// Browsers block this combo!

Wildcard + credentials is forbidden by the spec. And even without credentials, * opens your API to the entire internet.

Wrong Fix: Disable CORS Extension

Browser extensions that "disable CORS" only work on YOUR machine. Your users won't have them. Fix the server instead.

Speaker Notes: DEMO: Show the actual CORS error in DevTools. Open a blank page, open console, type: fetch('https://api.github.com/users/octocat') — this works because GitHub allows it! Now try fetch('https://www.google.com') — CORS error! Show the error message. "See how the error says 'No Access-Control-Allow-Origin header'? That means Google's server didn't send a CORS header allowing your origin." Walk through the fixes: (1) cors npm package is the standard for Express. (2) Dev proxy is great during development — the proxy runs on the same origin so there's no CORS. (3) The wrong fixes: "I've seen students install a Chrome extension to disable CORS. That only works on your machine. Your deployed app will still be broken."
!

Why * With Credentials is Dangerous

The Security Reason Behind the Restriction

The Attack Scenario

1. User is logged into bank.com (has session cookie)
↓
2. User visits evil.com
↓
3. evil.com runs: fetch('bank.com/api/balance', { credentials: 'include' })
↓
4. If bank allows * with credentials...
↓
5. evil.com reads your bank balance!

The Browser's Protection

Browsers refuse to honor Access-Control-Allow-Origin: * when credentials are included.

The server must explicitly name the allowed origin:

// Correct
Access-Control-Allow-Origin: https://myapp.com
Access-Control-Allow-Credentials: true

// Blocked by browser
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true

Best Practice

Keep an allowlist of origins on your server. Only return the origin in the header if it's on the list.

Speaker Notes: This connects everything together. "If a bank's API said 'allow any origin to read responses with credentials included,' then any website could read your bank data while you're logged in." The browser prevents this by spec. The lesson: "In production, always set the exact origin(s) you expect. Never use * when credentials are involved." For students' final projects: if your React app is on vercel-app.com and your API is on render.com, set Allow-Origin to your exact Vercel URL.

  • ✓ Origin = protocol + domain + port — all three must match
  • ✓ Same-Origin Policy blocks cross-origin reads by default

  • ✓ Fix CORS on the server, not with browser extensions
