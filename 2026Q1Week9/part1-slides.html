What is OWASP?

The Open Worldwide Application Security Project

A Non-Profit Foundation

OWASP produces free, open resources for web application security. Think of it as the "security bible" for web developers.

What They Produce

  • • The Top 10 security risks list
  • • Free testing tools (ZAP proxy)
  • • Cheat sheets & guidelines
  • • Community-driven & updated regularly

Why Should You Care?

  • ✗ Data breaches cost companies millions
  • ✗ Frontend is the #1 attack surface
  • ✗ Users trust YOUR interface with their data
  • ✗ "Someone else handles security" is a myth

Job Market Reality

Security awareness is increasingly expected in frontend job interviews.

Speaker Notes: OWASP is a non-profit that's been around since 2001. Everything they produce is free. The Top 10 list is updated every few years based on real-world data from hundreds of organizations. Emphasize: "Security is not just a backend problem. The frontend is where users interact with your app — it's the attack surface." Mention that interviewers at companies like Google, Meta, etc. will ask about XSS and CORS.

#1: Broken Access Control

The Frontend Developer's Biggest Misconception

The Wrong Way

// "Hiding" the admin button
{user.role === 'admin' && (
  <button onClick={deleteAllUsers}>
    Delete All Users
  </button>
)}

// Attacker opens DevTools...
// ...calls the API directly
fetch('/api/users', {method:'DELETE'})

The Right Way

  • ✓ Hide UI for better UX — that's fine
  • ✓ Always enforce on the server
  • ✓ Every API endpoint checks permissions
  • ✓ Never trust what the client sends

Real World

In 2019, a major bank's mobile app hid admin features in the UI — but anyone could call the API directly. Millions of records exposed.

Hiding a button is UX, not security. The server must ALWAYS check permissions.

Speaker Notes: This is the #1 mistake frontend devs make. Show the code on the left — "This looks reasonable, right? We only show the delete button to admins." Then reveal the problem: "Anyone can open DevTools and call fetch() directly. The button doesn't matter." Ask students: "If you remove a link from a nav bar, can users still visit that URL directly?" (Yes!) The lesson: frontend controls are for UX, not security. Always validate on the server.

#3 Injection & #5 Misconfiguration

Two More Risks Frontend Devs Enable

#3: Injection

Injection happens when untrusted data is sent as part of a command.

  • • SQL Injection: Malicious input in a database query
  • • HTML Injection: Injecting markup via user input
  • • XSS: Injecting JavaScript (our next topic!)

The Frontend Connection

Your frontend is where user input enters the system. Sanitizing early prevents injection downstream.

#5: Security Misconfiguration

Default or missing security settings:

  • ✗ Detailed error messages shown to users
  • ✗ Missing security headers (CSP, X-Frame-Options)
  • ✗ Default admin credentials left in place
  • ✗ Directory listing enabled on web server
  • ✗ CORS set to allow all origins

Your Job

Never show stack traces in production. Set proper security headers. We'll cover CSP and CORS in depth today.

Speaker Notes: Injection is #3 but students already know some of this from Week 3 (innerHTML). Emphasize that the frontend is the "front door" — all user input enters through your code. For misconfiguration: "How many of you have seen a full stack trace on a website error page?" That's misconfiguration. It gives attackers information about your tech stack. We'll see specific examples of proper headers (CSP) and CORS configuration later today.
