XSS

What is XSS?

Injecting Malicious Scripts Into Trusted Websites

The Attack

An attacker injects malicious JavaScript into a web page that other users visit. The victim's browser runs the script because it trusts the website.

What Attackers Can Do

  • ✗ Steal session cookies & auth tokens
  • ✗ Redirect users to phishing sites
  • ✗ Log keystrokes (capture passwords)
  • ✗ Modify the page content (fake forms)
  • ✗ Make API requests as the victim

How It Works

1. Attacker finds input that renders on page
↓
2. Injects <script> tag or JS event handler
↓
3. Victim visits the page normally
↓
4. Browser executes the malicious script
↓
5. Attacker gets cookies, data, or control
Speaker Notes: Key concept: the browser TRUSTS the website. So if evil.com can get its JavaScript to run on yourbank.com, the browser gives that script full access to yourbank.com's cookies, DOM, and APIs. Ask students: "Remember innerHTML from Week 3? That's exactly how XSS works." Emphasize that the attacker doesn't hack the server — they hack the PAGE that other users see.

Three Types of XSS

Stored, Reflected, and DOM-Based

Stored XSS

Malicious script is saved in the database (e.g., a comment, profile name).

Every user who views the page gets hit.

Comment: "Nice post!
<script>steal(cookies)</script>"

Most dangerous type

Reflected XSS

Malicious script is in the URL and reflected back by the server.

Victim must click a crafted link.

site.com/search?q=
<script>steal(cookies)</script>

Requires social engineering

DOM-Based XSS

Script executes via client-side JavaScript manipulating the DOM.

Never touches the server.

el.innerHTML =
location.hash.slice(1)

Frontend developer's problem

DOM-based XSS is 100% a frontend bug — it happens entirely in YOUR JavaScript code.

Speaker Notes: Walk through each type. Stored: "Imagine a forum where someone posts a comment containing a script tag. Now everyone who views that thread runs the attacker's code." Reflected: "The attacker emails you a link like site.com/search?q=<script>... — you click it, the server echoes your search term back, and the script runs." DOM-based: "This is the one that's YOUR problem as a frontend dev. Remember innerHTML? If you set innerHTML to something from the URL hash or user input, you've created a DOM-based XSS vulnerability." Emphasize that DOM-based XSS never hits the server — a WAF (Web Application Firewall) can't protect against it.

Live Demo: DOM-Based XSS

Building on What We Learned in Week 3

The Vulnerable Code

// A "search results" page
const query = new URLSearchParams(
  window.location.search
).get('q');

// DANGER: innerHTML with user input!
document.getElementById('results')
  .innerHTML = `Showing results for:
  <strong>${query}</strong>`;

// Attacker crafts URL:
// ?q=<img src=x onerror="alert(
//   document.cookie)">

The Safe Version

// Same page, done safely
const query = new URLSearchParams(
  window.location.search
).get('q');

// SAFE: textContent escapes HTML
const el = document.getElementById(
  'results'
);
el.textContent =
  `Showing results for: ${query}`;

// <img> tag is displayed as text,
// NOT executed as HTML!

Demo: Open DevTools console → Create a div → Set innerHTML vs textContent with <img src=x onerror="alert(1)">

Speaker Notes: LIVE DEMO INSTRUCTIONS: Open any webpage. Open DevTools console. Type: let div = document.createElement('div'); document.body.appendChild(div); Then show the dangerous version: div.innerHTML = '<img src=x onerror="alert(document.cookie)">' — the alert fires! Now show the safe version: div.textContent = '<img src=x onerror="alert(document.cookie)">' — it just displays as text. Ask students: "Which one did we learn about in Week 3?" (innerHTML). "Now you know WHY textContent is safer." Point out the onerror trick — attackers don't even need a script tag. Image tags, SVGs, event handlers all work.

Stored XSS: The Most Dangerous

One Malicious Input, Thousands of Victims

The Attack Scenario

1. Attacker submits comment with script
↓
2. Server saves comment to database
↓
3. Other users load the page
↓
4. Server sends comment (with script) to all
↓
5. Script runs in every victim's browser

Real World Examples

MySpace Worm (2005)

Samy Kamkar's XSS worm spread via profile pages. Added "Samy is my hero" to 1 million profiles in 20 hours.

British Airways (2018)

Injected script on payment page skimmed credit card details of 380,000 customers. $230M fine.

eBay (2015-2016)

XSS in listing descriptions redirected buyers to phishing pages.

Speaker Notes: Stored XSS is the worst because the attacker only needs to inject once, and it hits EVERY user. Tell the Samy worm story — it's entertaining and educational. A teenager wrote a worm that exploited XSS in MySpace profiles. Within 20 hours it had infected over 1 million profiles. The British Airways example shows the financial stakes: $230M fine for a single XSS-enabled breach. Ask: "Where in YOUR apps do users submit content that gets displayed to others? Comments? Usernames? Profile bios? Those are all potential Stored XSS vectors."

XSS in React

How React Protects You (Mostly)

React Auto-Escapes JSX

// React AUTOMATICALLY escapes this
function Comment({ text }) {
  // Even if text contains <script>...
  // React renders it as plain text!
  return <p>{text}</p>;
}

// text = "<script>alert('xss')</script>"
// Renders as: &lt;script&gt;alert...
// SAFE! The script is shown as text.

This is why XSS dropped from #3 to #7 in the OWASP list. Modern frameworks escape by default.

The Escape Hatch

// dangerouslySetInnerHTML
// The name is a WARNING!
function RichContent({ html }) {
  return (
    <div
      dangerouslySetInnerHTML={
        { __html: html }
      }
    />
  );
}

// If html comes from user input...
// YOU JUST CREATED AN XSS VULN!

Rule: Never use dangerouslySetInnerHTML with user-supplied data without sanitizing first.

Speaker Notes: "Good news — React protects you by default!" Show the left side: any value in JSX curly braces is automatically escaped. Even if someone enters a script tag as their username, React shows it as text, not HTML. "But React gives you an escape hatch — and they literally named it dangerouslySetInnerHTML to WARN you." The name has 'dangerous' in it for a reason. Ask students: "When might you actually need this?" (Rich text editors, rendering markdown, CMS content). Those cases need sanitization first. Mention that Vue has a similar escape hatch: v-html.
DEF

XSS Prevention Toolkit

Your Defense-in-Depth Strategy

1. Use textContent, Not innerHTML

For plain text, always use textContent. It automatically escapes HTML.

2. Trust Your Framework

React, Vue, Angular all auto-escape by default. Don't bypass their protections unless absolutely necessary.

3. Sanitize When You Must Render HTML

import DOMPurify from 'dompurify';

// Strips dangerous tags & attributes
const clean = DOMPurify.sanitize(
  userHTML
);

4. Content Security Policy (CSP)

HTTP header that blocks inline scripts entirely. Our next topic!

5. Validate & Encode Input

Validate on the client for UX. Always validate on the server for security. Encode special characters on output.

6. HttpOnly Cookies

Even if XSS happens, HttpOnly cookies can't be read by JavaScript. We'll cover this in Part 6.

Speaker Notes: This is the "defense in depth" approach. No single defense is perfect, so you layer them. Walk through each: (1) You already know this from Week 3. (2) Let React do its job — don't use dangerouslySetInnerHTML. (3) When you MUST render HTML (rich text editor, markdown), use DOMPurify to strip dangerous elements. It's the industry standard library. (4) CSP is next — it's like a firewall for the browser. (5) Client-side validation is for UX (showing "invalid email" immediately). Server-side validation is for security. (6) HttpOnly cookies are a last line of defense — even if XSS succeeds, the attacker can't steal the session cookie.
